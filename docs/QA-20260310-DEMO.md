# Run-Thru Matrix (8 Scenarios)

## Run-Thru 1 — Happy path 60-min booking (end-to-end)

**Covers: RRS-2 HOLD CREATED, RRS-3 VALID, RRS-5 writeback, request COMPLETE**

**Backend checks:**
- Booking Request: HOLD CREATED + Held Slot(s)/Summary + token/expiry/code
- Slot: HELD with matching token/email
- Booking: AWAITING CONFIRMATION then CONFIRMED
- Confirmation: Result VALID
- Slot: BOOKED + hold cleared
- Master: overwritten fields set
- Request: COMPLETE

⸻

## Run-Thru 2 — 30+30 booking

**Covers: 30-min branch, two-slot booking, multi-record writeback**

**Backend checks:**
- Held Slot(s) contains 2 links
- Booking Total Minutes = 60
- Both master records updated

⸻

## Run-Thru 3 — Invalid email

**Covers: FAILED - INVALID EMAIL email branch**

**Backend checks:**
- Request status FAILED - INVALID EMAIL
- No held slot(s)
- No booking created (or no slot held)

⸻

## Run-Thru 4 — NEEDS HELP

**Covers: NEEDS HELP branch + email**

**Backend checks:**
- Request status NEEDS HELP
- No held slots/bookings created

⸻

## Run-Thru 5 — Confirmation INVALID

**Covers: confirmation INVALID + email**

**Backend checks:**
- Confirmation Result INVALID
- Booking remains AWAITING CONFIRMATION
- Slot remains HELD

⸻

## Run-Thru 6 — Expiry + Confirmation EXPIRED

**Covers: RRS-4 expire sweep + confirmation EXPIRED branch**

**Backend checks:**
- Booking EXPIRED + Expired At set
- Request EXPIRED
- Slot AVAILABLE and hold fields cleared
- Confirmation Result EXPIRED

⸻

## Run-Thru 7 — Account credit request (normal) + reciprocal linking if possible

**Covers: RRS-6 NEW processing, evidence linking, NEW email**

**Backend checks:**
- Refund Status NEW
- Booking linked (if exists)
- Booking Request(s) linked
- If cancellation exists, reciprocal links set both ways

⸻

## Run-Thru 8 — Cancellation duplicate + Script 8 cancellation processing

**Covers: RRS-7 duplicate block, RRS-8 full revert, system notes, cancellation status transitions**

**Backend checks:**
- Duplicate cancellation → NOT PROCESSED - EXISTING REQ + email
- Approved cancellation processed:
- slots AVAILABLE + cleared booking link
- booking CANCELLED
- paid roster booking confirmation cleared
- master schedule reverted to AVAILABLE RESERVATION and reservation fields cleared
- cancellation status CANCELLED IN SYSTEM + System Notes audit

⸻

## Master Schedule Reconciliation Tests (Slot Sync)

**Run these after making edits to Master Schedule reserved blocks:**

	1.	Edit time of an AVAILABLE reserved block → new operational slot created, old becomes VOIDED
	
  2.	Delete an AVAILABLE reserved block → operational becomes VOIDED
	
  3.	Add a new reserved block → operational created
	
  4.	Modify/delete a BOOKED reserved block → operational stays NEEDS REVIEW (should not become VOIDED)
	
  5.	(Optional) Change Location string → verify ROOMS mapping behavior

⸻

- Triggering record ID
- Console logs / error
- Table + field values at failure point
- Expected vs actual outcome
